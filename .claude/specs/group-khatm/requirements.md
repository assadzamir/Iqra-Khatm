# Requirements: Group Khatm

> Generated by spec-engine v1.0.0

## Overview

Group Khatm is a collaborative Quran completion feature for the Iqra Islamic super-app (React Native + Expo SDK 52). An admin creates a Khatm group through a 5-step flow (Niyyah first), divides the 30 Juz among participants via a live 30-tile grid, and tracks collective progress in real time via Supabase Realtime. Participants can be existing Iqra users or people who join via an 8-character invite code shared manually by the admin. The feature includes Quran reader integration with auto-tracking, admin/co-admin role management, configurable push notification reminders, and a Khatm completion ceremony screen.

## Depends On

- Existing Iqra Quran reader (navigation param integration via `khatmContext`)
- Supabase Auth (user authentication)
- Supabase Realtime (live grid updates)
- expo-notifications (push notifications)
- @gorhom/bottom-sheet (Juz tile detail sheets)

## User Stories

### US-1: Create a Group Khatm (5-Step Flow)

**As an** authenticated Iqra user
**I want** to create a Group Khatm through a guided 5-step flow starting with Niyyah (intention)
**So that** I can organize a collective Quran reading with a clear spiritual purpose and practical configuration

#### Acceptance Criteria (EARS Notation)

1. WHEN the user taps "Create Khatm" from the home dashboard
   THE SYSTEM SHALL display Step 1 (Niyyah) with occasion type chips (GENERAL, MEMORIAL, RAMADAN, EID, SHIFA, CUSTOM), an optional dedicated-to name field (varchar 200), an optional relationship field (varchar 100), an optional free-text intention field (300 chars), and the Arabic banner text in Amiri font with teal (#117A7A) color

2. WHEN the user selects an occasion type and taps "Next"
   THE SYSTEM SHALL navigate to Step 2 (Group Details) displaying title field (varchar 80, required), language chips (AR/EN/UR/TR/FR/ID/MS), start date picker, and end date picker

3. WHEN the user fills the required title, selects start and end dates, and taps "Next"
   THE SYSTEM SHALL navigate to Step 3 (Assignment Rules) displaying assignment mode selector (ADMIN or PARTICIPANT) and a toggle for allowing multiple assignees per Juz (max 2 in v1, stored as `max_per_juz`)

4. WHEN the user configures assignment rules and taps "Next"
   THE SYSTEM SHALL navigate to Step 4 (Reminder Schedule) displaying default reminder windows (days before end date) with controls to add or remove windows

5. WHEN the user completes Step 4 and taps "Next"
   THE SYSTEM SHALL navigate to Step 5 (Review) displaying a summary of all settings from Steps 1 through 4

6. WHEN the user taps "Create" on Step 5
   THE SYSTEM SHALL call `supabase.rpc('generate_invite_code')`, insert a row into `khatm_groups`, insert the creator as ADMIN participant, display a full-screen Bismillah overlay for 2 seconds, then navigate to the main Khatm screen

7. WHEN the user taps "Back" on any step (2 through 5)
   THE SYSTEM SHALL navigate to the previous step with all previously entered data preserved

8. WHEN the user attempts to proceed from Step 2 without entering a title
   THE SYSTEM SHALL display an inline validation error "Title is required" below the title field and prevent navigation to Step 3

9. WHEN the user attempts to set an end date that is the same as or earlier than the start date
   THE SYSTEM SHALL display an inline validation error "End date must be after start date" and prevent navigation to Step 3

10. WHEN the `generate_invite_code()` RPC fails after 10 collision retries
    THE SYSTEM SHALL display an error toast "Could not create group. Please try again." and remain on Step 5

11. [inferred] WHEN the network request to insert the group fails
    THE SYSTEM SHALL display an error toast "Failed to create group. Check your connection and try again." and remain on Step 5 with all form data preserved

12. THE SYSTEM SHALL render the Bismillah overlay text in Amiri font, teal (#117A7A) color, centered on screen

13. THE SYSTEM SHALL render all Arabic text in the creation flow with both `textAlign: 'right'` AND `writingDirection: 'rtl'`

### US-2: View and Interact with the Main Khatm Screen

**As a** Khatm group member (any role)
**I want** to see the Khatm dashboard with Bismillah header, niyyah, collective progress, and the 30-tile Juz grid
**So that** I can monitor our collective reading progress and take action on my assigned Juz

#### Acceptance Criteria (EARS Notation)

1. WHEN a member opens the Khatm screen
   THE SYSTEM SHALL display the Bismillah header in Amiri font teal (#117A7A) centered, the Khatm title, the niyyah/dedication as a pull-quote, and a collective progress bar showing the percentage of completed Juz out of 30

2. THE SYSTEM SHALL render the Juz grid as 5 columns by 6 rows (30 tiles total)

3. WHILE the Khatm screen is open
   THE SYSTEM SHALL subscribe to Supabase Realtime on `khatm_juz_assignments` filtered by `group_id` and update tile visual states within 2 seconds of any database change

4. THE SYSTEM SHALL display each Juz tile with the Juz number in DM Sans bold and the Arabic opening words in Amiri font with `textAlign: 'right'` and `writingDirection: 'rtl'`

5. THE SYSTEM SHALL render tiles in the following visual states: Open (white/light background, teal border outline), Assigned (teal tint #D7F2E5, person initial badge bottom-right), In Progress (teal circular progress ring proportional to `progress_percent`), Completed (solid teal #117A7A, checkmark icon)

6. THE SYSTEM SHALL render the teal circular progress ring ONLY on tiles with status IN_PROGRESS

7. IF a Juz has 2 assignees
   WHEN the tile renders
   THE SYSTEM SHALL display a stacked double initial badge

8. THE SYSTEM SHALL display a collapsible Members section with an explicit show/hide toggle button

9. THE SYSTEM SHALL display a collapsible Juz grid section with an explicit show/hide toggle button

10. WHEN the Supabase Realtime subscription fails or disconnects
    THE SYSTEM SHALL display a "Reconnecting..." indicator and automatically retry the subscription

2. WHERE the summary card identifies a stalled Juz
   WHEN the admin taps the stalled Juz reference in the summary

### US-4: Juz Tile Bottom Sheet Interaction

**As a** Khatm group member
**I want** to tap a Juz tile to see details and take actions (assign, reassign, update progress, remind)
**So that** I can manage Juz assignments and track reading progress

#### Acceptance Criteria (EARS Notation)

1. WHEN a member taps a Juz tile
   THE SYSTEM SHALL open a `@gorhom/bottom-sheet` at the half-screen snap point displaying Juz number, Surah name in Arabic (Amiri, RTL) and English, current assignee(s), and progress percentage

2. WHERE the user's role is ADMIN or CO_ADMIN
   WHEN the bottom sheet opens
   THE SYSTEM SHALL display tabbed actions: Assign, Reassign, Progress, and Remind

3. WHERE the user's role is PARTICIPANT and the Juz is assigned to them
   WHEN the bottom sheet opens
   THE SYSTEM SHALL display the Progress tab and a "Start Reading" button

4. WHEN the user taps the Assign tab
   THE SYSTEM SHALL expand the bottom sheet to the full-screen snap point and display existing group members as selectable chips (name + initial) with a "+ New Person" chip at the end

5. WHEN the user selects an existing group member chip for assignment
   THE SYSTEM SHALL auto-fill the member's contact details with zero additional typing required

6. WHEN the user taps the "+ New Person" chip
   THE SYSTEM SHALL display a name field, a phone/email field, a message preview, and an "Edit message" toggle that reveals an editable notification text input

7. WHEN the user confirms a Juz assignment
   THE SYSTEM SHALL insert a row into `khatm_juz_assignments`, insert an audit log entry with action_type 'JUZ_ASSIGNED', auto-close the bottom sheet after 1.5 seconds, and animate the tile to its new visual state via Realtime

8. WHEN the user taps "Assign without notifying"
   THE SYSTEM SHALL complete the assignment without sending a push notification to the assignee

9. WHEN the user opens the Progress tab
   THE SYSTEM SHALL display a slider from 0 to 100 percent initialized to the current `progress_percent` value

10. WHEN the user adjusts the progress slider and confirms
    THE SYSTEM SHALL insert a row into `khatm_progress_updates` (recording `previous_percent`) and update `khatm_juz_assignments.progress_percent`

11. IF the user's role is ADMIN or CO_ADMIN
    WHEN they update another participant's progress via the Progress tab
    THE SYSTEM SHALL record the `khatm_progress_updates` row with `source: 'ADMIN_OVERRIDE'` and display a "Recorded by Admin" badge in the progress history

12. WHEN the user taps the Remind tab and confirms sending
    THE SYSTEM SHALL send an immediate push notification to the assigned participant for that Juz

13. WHEN an assignment insert fails because the Juz already has `max_per_juz` assignees
    THE SYSTEM SHALL display an error message "This Juz already has the maximum number of assignees" and keep the bottom sheet open

14. [inferred] WHEN the network request to update progress fails
    THE SYSTEM SHALL display an error toast "Failed to save progress. Please try again." and revert the slider to its previous value

### US-5: Quran Reader Integration with Auto-Tracking

**As a** participant assigned to a Juz
**I want** to tap "Start Reading" and have my reading progress automatically tracked as I read in the Iqra Quran reader
**So that** my progress updates for the group without manual input

#### Acceptance Criteria (EARS Notation)

1. WHEN an assigned participant taps "Start Reading" on their Juz tile bottom sheet
   THE SYSTEM SHALL set `activeReadingContext` in the Zustand store to `{ groupId, participantId, juzNumber, startPage, endPage }` (from `JUZ_PAGE_RANGES`) and navigate to the Iqra Quran reader at `JUZ_PAGE_RANGES[juzNumber].startPage`

2. WHILE `khatmContext` navigation param is present in the Quran reader
   THE SYSTEM SHALL calculate progress as `Math.round(((currentPage - startPage) / (endPage - startPage + 1)) * 100)`

3. WHILE the participant reads within their assigned page range with `khatmContext` active
   THE SYSTEM SHALL write progress to `khatm_juz_assignments.progress_percent` via a `khatm_progress_updates` insert with `source: 'AUTO_TRACKING'`, throttled to trigger on every 2-page advance OR when the user exits the reader

4. WHEN `currentPage >= endPage` during auto-tracked reading
   THE SYSTEM SHALL set `progress_percent` to 100 (which triggers the database trigger to mark the Juz as COMPLETED with `completed_at = now()`)

5. WHEN the participant exits the Quran reader while `activeReadingContext` is non-null
   THE SYSTEM SHALL write the final progress value to the database and then set `activeReadingContext` to null in the Zustand store

6. THE SYSTEM SHALL NOT modify any Quran reader behavior when the `khatmContext` navigation param is absent

7. THE SYSTEM SHALL display an "Auto-tracked" badge in the progress history for entries with `source: 'AUTO_TRACKING'`

8. [inferred] WHEN a throttled progress write fails due to network error
   THE SYSTEM SHALL queue the progress update locally in MMKV and retry on the next successful network write

### US-7: Push Notifications

**As a** Khatm participant
**I want** to receive push notifications for assignments, reminders, and milestones
**So that** I stay accountable and engaged with my reading commitment

#### Acceptance Criteria (EARS Notation)

1. WHEN a Juz is assigned to a participant (and the assignment is not marked "without notifying")
   THE SYSTEM SHALL send a push notification to the assigned participant containing the Juz number and group name

2. IF a Juz assignment has `status: 'ASSIGNED'` and `progress_percent: 0`
   WHEN 3 days have passed since `assigned_at`
   THE SYSTEM SHALL send a push reminder to the assigned participant

3. IF a Juz assignment has `status: 'IN_PROGRESS'`
   WHEN 4 days have passed since `last_updated_at` with no new progress update
   THE SYSTEM SHALL send a push reminder to the assigned participant

4. WHEN the current date matches `end_date - days_before` for any active row in `khatm_reminder_schedules`
   THE SYSTEM SHALL send a push notification to all group members whose assigned Juz are not yet COMPLETED

5. WHEN any single Juz assignment reaches `progress_percent: 100`
   THE SYSTEM SHALL send a celebratory push notification to all group members

6. WHEN all 30 Juz reach COMPLETED status (group status becomes COMPLETED)
   THE SYSTEM SHALL send a Khatm completion push notification to all group members

7. THE SYSTEM SHALL NOT send duplicate push notifications of the same type to the same participant for the same triggering event

8. [inferred] WHEN a push notification delivery fails
   THE SYSTEM SHALL log the failure and retry up to 3 times with exponential backoff

### US-8: Co-Admin Management

**As an** admin
**I want** to promote participants to Co-Admin and manage their permissions and removal
**So that** I can delegate group management responsibilities to trusted members

#### Acceptance Criteria (EARS Notation)

1. WHEN the admin taps "Promote to Co-Admin" on a JOINED participant in the members list
   THE SYSTEM SHALL update the participant's `role` to 'CO_ADMIN' and insert an audit log entry with action_type 'ROLE_ASSIGNED'

2. WHILE a participant has the CO_ADMIN role
   THE SYSTEM SHALL allow them to assign Juz, record proxy progress (source: ADMIN_OVERRIDE), send reminders, and manage invites

3. THE SYSTEM SHALL NOT allow a Co-Admin to delete the group, remove the admin, or promote other participants to Co-Admin

4. WHEN the admin taps "Remove Co-Admin" on a co-admin participant
   THE SYSTEM SHALL display a prompt: "Keep progress records attributed to [name]?" with YES and NO options

5. IF the admin selects YES on the co-admin removal prompt
   WHEN the removal is confirmed
   THE SYSTEM SHALL downgrade the participant's role to 'PARTICIPANT' and retain all `khatm_progress_updates` rows attributed to the former co-admin's `participant_id`

6. IF the admin selects NO on the co-admin removal prompt
   WHEN the removal is confirmed
   THE SYSTEM SHALL downgrade the participant's role to 'PARTICIPANT' and re-attribute all `khatm_progress_updates` rows where `source = 'ADMIN_OVERRIDE'` from the former co-admin's `participant_id` to the admin's `participant_id`

7. [inferred] WHEN the admin attempts to promote a participant whose status is 'REMOVED' or 'LEFT'
   THE SYSTEM SHALL display an error message "Cannot promote inactive members"

## Non-Functional Requirements

### Performance

1. THE SYSTEM SHALL render the 30-tile Juz grid within 300ms of receiving query data (measured as time from Supabase response to last tile painted on screen)

2. THE SYSTEM SHALL propagate Realtime tile state changes to all connected group members within 2 seconds of the database write

3. THE SYSTEM SHALL throttle auto-tracking progress writes to at most one write per 2-page advance, resulting in a maximum of 11 writes per Juz reading session for Juz 1 through 29 (approximately 20 pages each)

4. [inferred] THE SYSTEM SHALL complete the 5-step creation flow submission (Step 5 "Create" tap to Bismillah overlay appearance) within 3 seconds on a 3G network connection

5. [inferred] THE SYSTEM SHALL load the `useKhatmScreen(groupId)` query (group + all 30 Juz + participants) in a single Supabase request returning within 500ms on a 4G connection

### Security

6. THE SYSTEM SHALL enforce Row Level Security on all `khatm_*` tables such that group members can only read data for groups where they have a `khatm_participants` row with `status: 'JOINED'`

7. THE SYSTEM SHALL prevent role escalation by using split UPDATE policies on `khatm_participants`: one for admin (any field) and one for self-update (non-role fields only, with role equality check in WITH CHECK clause)

8. THE SYSTEM SHALL write `khatm_audit_log` entries exclusively via server-side operations (Edge Functions or database triggers) with no client-side INSERT policy

9. THE SYSTEM SHALL NOT expose internal database UUIDs in user-facing invite URLs; only the 8-character `invite_code` shall appear in shareable links

10. [inferred] THE SYSTEM SHALL validate all user inputs (title max 80 chars, intention max 300 chars, progress 0-100, dates, Juz number 1-30) both client-side before submission and server-side via database CHECK constraints

### Accessibility

11. THE SYSTEM SHALL provide press feedback on all tappable elements using `active:scale-[0.98]` with 300ms transitions

12. [inferred] THE SYSTEM SHALL ensure all Juz tile visual states are distinguishable by means beyond color alone: completed tiles display a checkmark icon, in-progress tiles display a progress ring, assigned tiles display an initial badge, and open tiles display only a border outline

13. THE SYSTEM SHALL render all Arabic Quran text using the Amiri font with both `textAlign: 'right'` AND `writingDirection: 'rtl'` properties set

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Supabase Realtime subscription drops silently, leaving stale Juz grid | Medium | High | Implement auto-reconnection with "Reconnecting..." UI indicator; add periodic poll fallback every 30 seconds when subscription is down |
| Invite code collision despite 30^8 keyspace (~6.5 x 10^11 combinations) | Low | Medium | `generate_invite_code()` retries up to 10 times before raising exception |
| Auto-tracking progress writes create excessive database load in large groups | Medium | Medium | Throttle writes to 2-page intervals; batch final write on reader exit; use single RPC for progress update |
| Push notification delivery failures via expo-notifications | Medium | Medium | Retry up to 3 times with exponential backoff; log all failures for admin visibility in audit log |
| Co-admin progress re-attribution (NO option) causes confusion in progress history | Low | Medium | Store original `actor_participant_id` in `khatm_audit_log` before re-attribution for full traceability |
| Deep link not handled on devices without Iqra installed | High | Medium | Universal link fallback to app store; admin sharing instructions in invite section |
| Du'a Khatm al-Quran content requires scholar review per supported language | Medium | High | Ship AR and EN content in v1; gate other languages behind content-ready flag in JSON asset |
| Khatm cycle creation with many participants causes timeout | Low | Medium | Use a single Supabase RPC that inserts group + copies participants in one transaction |
| RLS policy recursion if helper functions are not SECURITY DEFINER | Medium | High | All RLS helper functions (`my_participant`, `is_admin_or_coadmin`, `is_admin`) declared SECURITY DEFINER |

## Out of Scope

- WhatsApp/SMS/Email automated sending via Twilio (push notifications only in v1)
- HifzAI memorization progress connection to Khatm (`source: AUTO_TRACKING` reserved in schema)
- Sub-Juz (half-Juz) assignments (use `max_per_juz = 2` as workaround)
- Web-only join form (Iqra app download required to participate)
- Broadcast messaging to group members
- AI-powered admin summaries (data structure supports future Claude upgrade; v1 uses rules-based computation)
- `max_per_juz > 2` (v2 feature; schema `smallint` supports arbitrary values)
- AUTO and MIXED assignment modes (v1 supports ADMIN and PARTICIPANT only)
- PAUSED group status (v1 supports ACTIVE, COMPLETED, ARCHIVED only)
- Privacy sharing toggle (`privacy_share_progress` column reserved but not enforced in v1)

### Lessons Applied

The following lessons from the prior Group-khatm monorepo build influenced these requirements:

1. **Every UI state flag must have an explicit setter** (Lesson 1) -- Influenced US-2 AC-8/AC-9 requiring explicit toggle buttons for collapsible sections, and US-5 AC-1/AC-5 requiring explicit set/clear of `activeReadingContext` in the Zustand store.

2. **Derive all types from actual schema first** (Lesson 2) -- The database schema in this spec is the authoritative source; all TypeScript types must derive from it, not the other way around.

3. **Fields set on first interaction must document initialization semantics** (Lesson 3) -- `started_at` is set by database trigger when `progress_percent` first exceeds 0; `completed_at` is set when `progress_percent` reaches 100; `activeReadingContext` is set on "Start Reading" tap and cleared on reader exit. All documented explicitly.

4. **React Native: use `globalThis` not `global`, `Buffer.from()` not `btoa`** (Lesson 4) -- Noted for implementation; mobile code must use `Buffer.from(x).toString('base64')`.

5. **Arabic text requires BOTH `textAlign: 'right'` AND `writingDirection: 'rtl'`** (Lesson 5) -- Called out in US-1 AC-13, US-2 AC-4, and NFR-13 as mandatory for every Arabic text element.

6. **AVAILABLE status does not exist as rows** (Research bug FIX-P2-5/6) -- US-11 specifies "unassigned" as absence of a row in `khatm_juz_assignments`, not a status value.

7. **Role escalation prevention via split UPDATE policies** (Research bug FIX-A2) -- NFR-7 requires split policies explicitly.

8. **RLS helper functions must be SECURITY DEFINER** (Research compilation) -- Risk register entry for recursive RLS; all helpers specified as SECURITY DEFINER.
