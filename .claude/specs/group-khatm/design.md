## Architecture Overview

Group Khatm follows a client-server architecture with Supabase as the backend. The React Native mobile app (Expo SDK 52) communicates with Supabase Postgres via the JS client for CRUD operations, subscribes to Supabase Realtime for live grid updates, and uses expo-notifications for push. State management uses Zustand + MMKV for client state and TanStack Query for server state caching.

The feature is structured as a self-contained module within the Iqra app, integrating with the existing Quran reader via navigation params. All Group Khatm files reside under `src/features/khatm/`.

### File: `src/features/khatm/store.ts`

```typescript
import { create } from 'zustand';
import { createJSONStorage, persist } from 'zustand/middleware';
import { mmkvStorage } from '@/lib/mmkv'; // existing MMKV adapter in Iqra
import type { KhatmReadingContext } from './types';

interface KhatmStore {
  // -- Persisted state (MMKV) --
  activeGroupId: string | null;

  // -- Ephemeral UI state (accordion sections) --
  juzGridCollapsed: boolean;
  membersCollapsed: boolean;
  inviteJoinCollapsed: boolean;
  remindersCollapsed: boolean;
  activeReadingContext: KhatmReadingContext | null;

  // -- Explicit setters (Lesson 1: every flag needs a setter) --
  setActiveGroupId: (id: string | null) => void;
  setJuzGridCollapsed: (collapsed: boolean) => void;
  setMembersCollapsed: (collapsed: boolean) => void;
  setInviteJoinCollapsed: (collapsed: boolean) => void;
  setRemindersCollapsed: (collapsed: boolean) => void;
  setActiveReadingContext: (ctx: KhatmReadingContext | null) => void;
}

export const useKhatmStore = create<KhatmStore>()(
  persist(
    (set) => ({
      activeGroupId: null,
      juzGridCollapsed: false,
      membersCollapsed: true,
      inviteJoinCollapsed: true,
      remindersCollapsed: true,
      activeReadingContext: null,

      setActiveGroupId: (id) => set({ activeGroupId: id }),
      setJuzGridCollapsed: (collapsed) => set({ juzGridCollapsed: collapsed }),
      setMembersCollapsed: (collapsed) => set({ membersCollapsed: collapsed }),
      setInviteJoinCollapsed: (collapsed) => set({ inviteJoinCollapsed: collapsed }),
      setRemindersCollapsed: (collapsed) => set({ remindersCollapsed: collapsed }),
      setActiveReadingContext: (ctx) => set({ activeReadingContext: ctx }),
    }),
    {
      name: 'khatm-store',
      storage: createJSONStorage(() => mmkvStorage),
      partialize: (state) => ({
        activeGroupId: state.activeGroupId,
        // activeReadingContext NOT persisted -- ephemeral session state
        // accordion collapsed states NOT persisted -- reset on app restart
      }),
    }
  )
);
```

**Initialization semantics (Lesson 3):**
- `activeGroupId`: Set when user creates or opens a Khatm. Persisted across app restarts via MMKV. Cleared (`null`) when user has no active Khatm.
- `juzGridCollapsed`: Defaults to `false` (expanded). Reset on app restart. Set via `setJuzGridCollapsed(true/false)` on accordion header tap.
- `membersCollapsed`: Defaults to `true` (collapsed). Reset on app restart. Set via `setMembersCollapsed(true/false)` on accordion header tap.
- `inviteJoinCollapsed`: Defaults to `true` (collapsed). Reset on app restart. Set via `setInviteJoinCollapsed(true/false)` on accordion header tap. Auto-set to `false` when deep link contains `iqra://khatm/join/:code` so the join field is pre-expanded.
- `remindersCollapsed`: Defaults to `true` (collapsed). Reset on app restart. Set via `setRemindersCollapsed(true/false)` on accordion header tap.
- `activeReadingContext`: Set to `{ groupId, participantId, juzNumber, startPage, endPage }` when participant taps "Start Reading". Cleared to `null` when participant exits the Quran reader or when Juz is marked complete. NOT persisted -- if app crashes during reading, context is lost and user must tap "Start Reading" again. Progress already written to DB is preserved.

### GroupKhatmScreen (One Main Screen)

- **Purpose**: The single entry point for the Group Khatm feature. Contains: fixed Bismillah header with group name + settings gear + Create button; always-visible CollectiveProgressBar; AdminSummaryCard (admin/co-admin only); four collapsible accordion sections: Juz Grid, Members, Invite & Join, Reminders & Settings.
- **Covers**: US-1 (via CreateKhatmBottomSheet), US-2, US-3, US-4, US-5, US-6, US-7, US-8, US-10, US-11, US-12
- **File**: `src/features/khatm/screens/GroupKhatmScreen.tsx`
- **Interface**:
  - `GroupKhatmScreen: React.FC<NativeStackScreenProps<KhatmStackParamList, 'GroupKhatm'>>`
  - Params: `{ groupId?: string; joinCode?: string }` — `groupId` from navigation; `joinCode` from deep link auto-expands Invite & Join accordion
  - Uses `useKhatmScreen(groupId)` for data, `useKhatmRealtime(groupId)` for live updates
  - Reads all accordion collapsed states from `useKhatmStore`
  - Holds `BottomSheetModal` refs: `createSheetRef`, `settingsSheetRef`, `juzSheetRef`
- **Dependencies**: `JuzGrid`, `MembersSection`, `AdminSummaryCard`, `CollectiveProgressBar`, `CreateKhatmBottomSheet`, `JuzBottomSheet`, `GroupSettingsBottomSheet`, `useKhatmScreen`, `useKhatmRealtime`
- **Layout**:
  ```
  ScrollView (vertical)
  ├── Fixed header: Bismillah (Amiri) + group name + [gear icon] + [+ Create] button
  ├── CollectiveProgressBar  (always visible)
  ├── AdminSummaryCard       (admin/co-admin only, always visible)
  ├── AccordionSection: "Juz Grid"         (default: expanded)
  │     └── JuzGrid (30 tiles, 5×6 FlatList)
  ├── AccordionSection: "Members"          (default: collapsed)
  │     └── MembersSection
  ├── AccordionSection: "Invite & Join"    (default: collapsed)
  │     ├── Invite code display + copy button  (admin/co-admin)
  │     └── Join code entry field + button     (participant or new user)
  └── AccordionSection: "Reminders & Settings"  (default: collapsed)
        ├── Reminder schedule preview (next reminder date)
        └── "Open Settings" → opens GroupSettingsBottomSheet
  ```
- **Arabic text rule**: Bismillah banner and Niyyah pull-quote use Amiri font with `textAlign: 'right'` and `writingDirection: 'rtl'`

## API Contracts

All data access goes through the Supabase JS client (not custom REST endpoints). The contracts below describe the Supabase queries and RPCs used by the TanStack Query hooks.

### File: `src/features/khatm/hooks/useKhatmQueries.ts`

```typescript
// Query key factory
export const khatmKeys = {
  all: ['khatm'] as const,
  groups: () => [...khatmKeys.all, 'groups'] as const,
  screen: (groupId: string) => [...khatmKeys.all, 'screen', groupId] as const,
};

// -- useKhatmGroups --
// Covers: US-2 (group list on dashboard)
// Returns all groups where the current user is a JOINED participant
// Supabase query: khatm_participants.select('*, group:khatm_groups(*)').eq('user_id', userId).eq('status', 'JOINED')
export function useKhatmGroups(): UseQueryResult<KhatmGroupCard[]>;

interface KhatmGroupCard {
  group_id: string;
  title: string;
  role: ParticipantRole;
  completed_count: number;   // count of distinct juz_number with status COMPLETED
  member_count: number;
  start_date: string;
  end_date: string;
  status: GroupStatus;
}

// -- useKhatmScreen --
// Covers: US-2, US-3, US-4
// Single query returning full Khatm dashboard data
// Supabase query: khatm_groups.select('*, participants:khatm_participants(*), assignments:khatm_juz_assignments(*)').eq('id', groupId).single()
export function useKhatmScreen(groupId: string): UseQueryResult<KhatmScreenData>;

// -- useKhatmRealtime --
// Covers: US-2 AC-3
// Subscribes to postgres_changes on khatm_juz_assignments filtered by group_id
// On any INSERT/UPDATE/DELETE: invalidates khatmKeys.screen(groupId)
// Cleanup: supabase.removeChannel(channel) on unmount
export function useKhatmRealtime(groupId: string): void;
```

## Error Handling Strategy

### Error Taxonomy

| Error Class | Source | User-Facing Message | HTTP Status | Recovery |
|------------|--------|---------------------|-------------|----------|
| Validation | Client-side form checks | Specific field error (e.g., "Title is required") | N/A (client) | Fix input and retry |
| Validation | Supabase CHECK constraint | "Invalid value provided. Please check and try again." | 400 | Fix input and retry |
| Auth | Supabase Auth session expired | "Your session has expired. Please sign in again." | 401 | Redirect to login |
| Authorization | RLS policy denial | "You don't have permission for this action." | 403 | Contact group admin |
| Not Found | Invalid invite code lookup | "Invalid code. Please check and try again." | 404 | Re-enter code |
| Conflict | max_per_juz exceeded | "This Juz already has the maximum number of assignees." | 409 | Choose different Juz |
| Conflict | Duplicate group membership | "You are already a member of this Khatm." | 409 | Navigate to existing group |
| Network | Supabase unreachable | "Connection error. Check your internet and try again." | N/A | Auto-retry with backoff |
| Realtime | Subscription dropped | "Reconnecting..." (subtle indicator) | N/A | Auto-reconnect |
| Internal | Unexpected Supabase error | "Something went wrong. Please try again." | 500 | Retry; log to console |

### Error Propagation

- **Frontend (React Native)**: Errors displayed via toast notifications (using existing Iqra toast system). Form validation errors shown inline below the relevant field. Realtime disconnection shown as a subtle banner. Bottom sheet errors shown in-sheet without dismissing.
- **Supabase Client**: All Supabase calls wrapped in try/catch within TanStack Query mutation/query functions. Supabase client errors have `{ code, message, details, hint }` shape. The `onError` callback in mutations maps error codes to user messages.
- **Database Layer**: CHECK constraints return specific violation messages. RLS denials return 403-equivalent. Trigger failures (e.g., `check_group_completion`) are logged server-side. Unique constraint violations caught and mapped to Conflict errors.
- **Push Notifications**: Failures logged to `khatm_audit_log` with action_type context. Retried up to 3 times. Never block UI operations.

### User-Facing Error Message Strategy

All error messages:
1. Never expose SQL errors, table names, or column names
2. Never expose UUIDs or internal IDs
3. Always suggest an action the user can take
4. Use plain language (no technical jargon)

## State Management

### State Diagram -- Khatm Group Lifecycle

```
[Creating] --submit success--> [Active]
[Creating] --submit failure--> [Creating] (retry)
[Active]   --all 30 Juz done-> [Completed] (via trigger)
[Active]   --admin archives--> [Archived]
[Completed]--admin archives--> [Archived]
[Completed]--start new cycle-> [Active] (new group, cycle N+1)
```

### State Diagram -- Juz Assignment Lifecycle

```
(no row)     --assign-------> [Assigned]   (progress_percent = 0)
[Assigned]   --progress > 0-> [In Progress] (started_at set by trigger)
[In Progress]--progress=100-> [Completed]   (completed_at set by trigger)
[Completed]  --reassign----> [Assigned]    (new row, old row deleted or kept)
[Assigned]   --unassign----> (no row)      (row deleted)
```

### State Diagram -- Quran Reader Auto-Tracking

```
[Idle]           --"Start Reading" tap--> [Reading]
[Reading]        --page advance (2+)----> [Reading] (write progress, throttled)
[Reading]        --currentPage>=endPage-> [Complete] (write 100%, clear context)
[Reading]        --exit reader----------> [Idle] (write final progress, clear context)
[Reading]        --network error--------> [Reading] (queue locally, retry)
```

### State Storage

- **Where**: Zustand store (`useKhatmStore`) backed by MMKV for persistence. TanStack Query for server state cache.
- **Persistence**: `activeGroupId` survives app restart (MMKV). `juzGridCollapsed`, `membersCollapsed`, `activeReadingContext` reset on restart (ephemeral). TanStack Query cache survives for `staleTime` duration only.
- **Sync (Frontend-Backend)**: Supabase Realtime subscription invalidates TanStack Query cache on any `khatm_juz_assignments` change for the active group. Mutations use `onSuccess` to invalidate relevant query keys. Optimistic updates NOT used in v1 to avoid consistency issues with multi-user Realtime -- always wait for server confirmation.

## Alternatives Considered

### Alternative 1: WebSocket-based custom server instead of Supabase Realtime

- **Description**: Run a custom WebSocket server (e.g., Socket.io on Node.js) for real-time grid updates
- **Pros**: Full control over message format and delivery guarantees
- **Cons**: Additional infrastructure to maintain, more deployment complexity, higher cost
- **Why rejected**: Supabase Realtime is included in the existing Supabase plan, requires zero additional infrastructure, and the `postgres_changes` subscription model fits the use case (react to row changes on `khatm_juz_assignments`)

### Alternative 2: Twilio WhatsApp for notifications in v1

- **Description**: Use Twilio API for WhatsApp/SMS notifications as in the prior monorepo build
- **Pros**: Reaches users who don't have the app; WhatsApp is ubiquitous in Muslim-majority countries
- **Cons**: Requires Twilio account + API approval for WhatsApp Business; cost per message; requires Edge Function for sending; potential for rate limiting
- **Why rejected**: Push notifications via expo-notifications are free and reach all Iqra app users. v1 requires app download to participate, so push covers 100% of active users. Twilio can be added in v2 for outreach to non-app contacts.

### Alternative 3: AUTO and MIXED assignment modes in v1

- **Description**: Include all four assignment modes from the prior monorepo (AUTO, ADMIN, PARTICIPANT, MIXED)
- **Pros**: More flexibility for different group dynamics
- **Cons**: AUTO mode requires complex round-robin logic with edge cases (what if no participants yet?); MIXED mode requires permission negotiation between admin-assigned and self-claimed Juz
- **Why rejected**: ADMIN and PARTICIPANT modes cover the two primary use cases. AUTO/MIXED can be added in v2 with the `assignment_mode` enum already supporting them in the schema.

### Alternative 4: Optimistic updates for progress writes

- **Description**: Update the UI immediately on progress slider change, roll back on server error
- **Pros**: Faster perceived performance for the user
- **Cons**: In a multi-user Realtime environment, optimistic updates can flash incorrect state before the Realtime event arrives with the server-confirmed value; creates a confusing "jump" in the progress ring
- **Why rejected**: Server-confirmed writes with Realtime propagation provide consistent state across all group members. The 2-second Realtime latency is acceptable for progress updates.

## Migration Plan

This is a greenfield feature. No existing database tables or functionality to migrate.

- **Database migrations**: New migration file `001_khatm_schema.sql` creates all `khatm_*` tables, enums, triggers, functions, indexes, and RLS policies. Run via `supabase db push` or `supabase migration up`.
- **Data backfilling**: None required. The prior Group-khatm monorepo is a separate project; no data migration between projects.
- **Backward compatibility**: The Quran reader receives an optional `khatmContext` navigation param. When absent, reader behavior is unchanged (US-5 AC-6). No other existing Iqra features are modified.
- **Rollback plan**: Drop all `khatm_*` tables and enums via a down migration. Remove `src/features/khatm/` directory. Remove Khatm card from home dashboard. No impact on other Iqra features.

## Implementation Notes (from lessons)

1. **`globalThis` not `global`; `Buffer.from(x).toString('base64')` not `btoa()`** -- All mobile code must use these React Native-safe alternatives. Edge Functions (Deno) may use `btoa()` safely.

2. **Barrel exports must use `export type` for interfaces** -- `src/features/khatm/index.ts` must split exports: `export { EnumName }` for enums used as values, `export type { InterfaceName }` for pure type exports, to satisfy `isolatedModules`.

3. **Arabic text rendering** -- Every component that renders Arabic text MUST set both `textAlign: 'right'` AND `writingDirection: 'rtl'`. Setting only one produces incorrect layout. This applies to: Bismillah header, Juz tile Arabic names, Du'a completion text, Niyyah banner.

4. **MMKV storage adapter** -- The Zustand store uses the existing Iqra MMKV adapter (assumed at `@/lib/mmkv`). If this path differs in the actual Iqra codebase, update the import in `store.ts`.

5. **Supabase client import** -- Use the existing Iqra Supabase client singleton (assumed at `@/lib/supabase`). Do not create a new client instance.
